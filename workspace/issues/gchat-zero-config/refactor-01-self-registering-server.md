## 🔁 Refactor

The MCP server currently requires `GC_AGENT_ID` as an external env var, generated by a separate CLI `join` command. This breaks because hook env vars don't propagate to the MCP server process. The server must own its own agent lifecycle: generate a fresh UUID on startup, register with StateService, join the project conversation, and clean up on shutdown.

## 🤝 Behavioral contract

- All 8 MCP tools continue to work identically
- Channel push notifications continue to work
- File-based shared state format is unchanged
- The only breaking change: `GC_AGENT_ID` is no longer required as an env var. It is generated internally.

## 📍 Current structure

```
src/constants/env.ts       — reads GC_AGENT_ID from process.env, exits if missing
src/index.ts               — imports GC_AGENT_ID, passes to poller, cleanup only stops poller
src/handlers/tool-handlers.ts — imports GC_AGENT_ID directly for send_message participant check
src/services/inbox-poller.ts  — polls inbox for new messages
src/cli.ts                 — generates agent UUID via registerAgent(), prints to stdout
```

The agent lifecycle is split across two processes (CLI and MCP server) with an env var bridge that doesn't work.

## 🎯 Target structure

```
src/constants/env.ts       — reads GC_PROJECT_PATH (optional) and GC_POLL_INTERVAL_MS (optional)
src/index.ts               — generates UUID, registers agent, joins project conversation,
                              passes agentId to handlers and poller,
                              full cleanup on shutdown (leave, unregister, stop poller)
src/handlers/tool-handlers.ts — receives agentId as parameter (no env import)
src/services/inbox-poller.ts  — receives agentId via start() parameter (unchanged API)
```

The agent lifecycle is fully owned by the MCP server process.

## ⚠️ Risks & rollback

- Breaking change: anyone using the old `GC_AGENT_ID` + hooks flow will need to switch to the new approach. This is intentional — the old flow was broken.
- Rollback: revert the commit. No data migration needed since the state format is unchanged.
- Lock scope inconsistency: registerAgent() uses withFileLock(agentsPath) while unregisterAgent() uses withStateLock. All mutations touching agents.json and conversations.json must use the same lock scope (withStateLock) to prevent data races.

## ✅ Acceptance criteria

- [ ] Server starts without `GC_AGENT_ID` env var
- [ ] Server generates a fresh UUID per startup
- [ ] Server registers as an agent and joins the project conversation on startup
- [ ] Server leaves conversations and unregisters on shutdown (SIGTERM, SIGINT, transport close)
- [ ] Project path determined by `GC_PROJECT_PATH` env var with `process.cwd()` fallback
- [ ] `handleToolCall` receives agentId as parameter, not from env import
- [ ] All existing tests pass
- [ ] `npm run build` succeeds

   - Remove `import { GC_AGENT_ID }` from env
   - Add `agentId: string` parameter to `handleToolCall`
   - Replace all `GC_AGENT_ID` references with the `agentId` parameter
4. Update `src/index.ts`:
   - Import `GC_PROJECT_PATH` from env (nullable)
   - In `main()`:
     a. After `stateService.init()`: call `const agent = await stateService.registerAgent(projectPath)`
     b. Use `agent.id` as the agentId for all downstream consumers
     c. Determine `const projectPath = GC_PROJECT_PATH ?? process.cwd()`
   - Pass `agentId` to `handleToolCall` in the CallToolRequestSchema handler
   - Pass `agentId` to `inboxPoller.start()`
   - In cleanup (SIGTERM, SIGINT, server.onclose):
     a. Write leave system messages and notifications for each conversation (iterate agent.conversations)
     b. Call `stateService.unregisterAgent(agentId)` — handles participant removal and archiving
     c. Call `inboxPoller.stop()`
5. Run `npm run build` — fix any compilation errors
6. Run `npm test` — fix any test failures
7. Manually verify: start server without GC_AGENT_ID, check agents.json, kill server, check cleanup
8. Confirm behavioral contract, tests, and acceptance criteria
