# 📘 As a developer, I want to install group-chat-mcp with one command so that my agents can communicate without manual configuration

## 🔗 Dependencies

- None

## 🔀 Related Issues

- workspace/issues/group-chat-mcp/feature-00-group-chat-mcp.md — original feature spec

---

## 🛠️ Skills, tools & MCPs

- `mcp-builder` — MCP server implementation patterns

---

## 👤 User Story

- As a 👨‍💻 Developer, I want to run `npm install -g group-chat-mcp && gchat install` and have multi-agent communication working immediately, so that I don't need to manually configure env vars, hooks, or MCP settings.

## 🚀 End Goal

- A developer installs the npm package globally, runs `gchat install`, selects their IDE (Claude Code / Cursor), and the MCP server is fully configured. No hooks, no env vars, no manual JSON editing. Each agent session auto-registers on connect and auto-cleans up on disconnect.

## 📍 Currently

- MCP server crashes without `GC_AGENT_ID` env var
- `GC_AGENT_ID` is generated by a separate `cli.ts join` command
- Users must manually configure hooks (SessionStart/SessionEnd) AND MCP server config
- Hook env vars don't propagate to the MCP server process — the architecture is broken

## 🎯 Should

- MCP server generates a fresh agent UUID on startup, registers itself, joins the project conversation, and cleans up on shutdown
- `gchat install` writes the correct MCP config to Claude Code or Cursor settings
- Zero manual configuration after running the installer

---

## ✅ Acceptance Criteria

- [ ] `npm install -g group-chat-mcp` makes `gchat` available globally
- [ ] `gchat install` writes valid MCP server config to the selected IDE's settings
- [ ] After gchat install, the IDE starts the MCP server using the written config without path resolution errors
- [ ] Starting the MCP server requires zero env vars (GC_AGENT_ID removed)
- [ ] Each MCP server instance generates a fresh UUID and auto-registers as an agent
- [ ] Server auto-joins the project conversation based on env var or cwd (one shared conversation per project path — all agents in the same directory share it)
- [ ] Server auto-leaves conversations and unregisters on disconnect/shutdown
- [ ] Shutdown notification logic reuses existing writeNotificationToParticipants from tool-handlers.ts (no duplication)
- [ ] Multiple MCP server instances in the same project folder each get unique agent IDs
- [ ] Stale agent reaping: on startup, server prunes agents.json entries whose process is no longer alive
- [ ] Concurrent registration: N simultaneous server startups result in exactly N entries in agents.json with no corruption
- [ ] Old `cli.ts` join/leave commands are removed
- [ ] README reflects the new installation flow
